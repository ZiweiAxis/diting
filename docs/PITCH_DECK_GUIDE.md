### Slide 5: eBPF - 核心技术亮点

**什么是 eBPF?**
> 在 Linux 内核中运行安全的"小程序"

**为什么用 eBPF?**
```
传统监控               eBPF
┌──────────┐           ┌──────────┐
│ 用户态   │           │ 内核态   │
│ strace  │  1000x    │ 零拷贝   │
│ ltrace  │   →      │ 原生执行  │
└──────────┘           └──────────┘
慢，性能差               快，性能优
```

**监控能力:**
- ✅ 系统调用
- ✅ 文件操作
- ✅ 网络连接
- ✅ 进程创建

**性能:**
- 延迟: < 1% 开销
- 吞吐: > 10k events/s
- 内存: < 10MB

### Slide 6: AI 意图分析

**问题:**
```
Agent 说: "帮我把昨天的生产日志删掉"

规则引擎: ?
  - "删除" = 危险？
  - "日志" = 敏感？
  - "昨天" = 时间范围？

  → 规则太多，难以覆盖
```

**AI 解决:**
```
LLM 分析:
  意图: 删除日志文件
  路径: /var/log/ (系统目录)
  影响: 可能影响故障排查
  风险: 高
  建议: 需要人工审批

  → 50 字智能判断
```

## 三、Q&A 准备

### Q1: 如果 Agent 绕过你们的代理怎么办？
**A:** "这正是 eBPF 的核心价值。我们的 eBPF 模块直接在内核层监控系统调用，即使 Agent 逃逸容器或绕过代理，仍然会被捕获和拦截。这是传统方案做不到的。"

### Q2: LLM 分析会不会太慢？
**A:** "我们做了两层优化：
> 第一，用 7B 本地小模型，响应时间 < 2 秒。
> 第二，对于超低延迟场景，可以纯规则引擎模式，延迟 < 10 毫秒。
> 
> 根据我们的测试，99% 的请求延迟 < 100 毫秒。"

### Q3: 支持哪些 AI 框架？
**A:** "与框架无关。只要 Agent 通过 HTTP/gRPC 通信，都可以接入。我们已经测试过 LangChain、AutoGPT、OpenClaw 等主流框架。"

### Q4: 市场竞争激烈，你们的优势是什么？
**A:** "AI 安全是新赛道，目前没有直接的竞品。
> 
> 传统安全公司做的是 Web 防火墙，不懂 AI。
> AI 公司做的是大模型，不懂安全。
> 
> 我们是唯一一家同时懂内核、懂 AI、懂安全的公司。"

### Q5: 如何获客？
**A:** "我们有三个渠道：
> 第一，开源核心引擎，吸引开发者。
> 第二，与云厂商合作，打包销售。
> 第三，通过技术社区和活动触达决策者。"
